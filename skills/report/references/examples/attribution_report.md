# Attribution Report Example

## 适用场景
- 用户重点想知道“为什么变了”
- 不是一般复盘，而是要解释主因、次因和贡献度
- 适合收入变化、份额变化、效率变化等问题

## 阅读目标
- 先定义现象
- 再分解原因
- 最后明确主因、次因和动作

## 开头写法
- 第一段必须先定义“变化现象”
- 不要先写方法或背景
- 要让读者明确知道：这份报告的任务是解释变化来源

```markdown
本期收入同比下降 11.3%，核心问题不是“是否下降”，而是“下降主要由什么驱动”。从当前分解结果看，订单量下降是首要因素，客单价走弱是次要因素，结构变化仅起到小幅缓冲作用。因此，这份报告的重点是区分主因和次因，而不是对所有因素平均展开。
```

## 证据块写法
- 证据块要能回答“主因是谁、次因是谁、影响大小如何”
- 不能只有数字，没有主次解释

```markdown
### 变化分解

| 因素 | 贡献度 |
| --- | --- |
| 订单量下降 | -9.1pp |
| 客单价下降 | -3.4pp |
| 结构变化 | +1.2pp |

判断：本次收入下滑的主因是订单量下降，其次才是客单价走弱。
解释：量的负向贡献显著高于价，说明当前问题重点不在定价，而在需求或渠道承接能力。
```

## 归因段落怎么写
- `现象定义`
- `变化分解`
- `主因分析`
- `次因分析`
- `行动建议`

## 正文与附件边界
- 正文：变化分解表、Top drivers、关键解释
- 附件：完整交叉分解、长周期拆分、补充明细

## 不要这样写
- 用“多因素共同影响”结束，不给主次关系
- 把所有驱动因素都堆成长段落
- 只给百分比，不解释这些变化意味着什么
- 让读者自己去附件找主因
