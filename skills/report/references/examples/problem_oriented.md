# Problem Oriented Example

## 适用场景
- 用户已经明确感知到“有问题”
- 希望知道问题在哪、范围多大、后面该先查什么
- 典型问题：下滑、异常、偏离、风险项

## 阅读目标
- 读者先看到“问题是什么”
- 再看到“问题集中在哪”
- 最后再进入“为什么”

## 开头写法
- 第一段必须把问题说清楚，不要从背景介绍开始
- 开头至少包含：问题对象、变化幅度、影响范围、是否需要继续诊断

```markdown
本次下滑不是全局性回落，而是集中发生在上海区域代理渠道。该渠道订单量同比下降 18.2%，同时华东重点产品转化率下降 4.1 个百分点，说明问题已从单点异常扩大为局部经营压力，需要进一步围绕渠道结构和重点产品承接能力展开诊断。
```

## 证据块写法
- 第一块证据用于“确认问题存在”
- 第二块证据用于“说明问题集中在哪”

```markdown
### 问题识别

| 问题点 | 异常表现 | 偏离程度 |
| --- | --- | --- |
| 上海区域代理渠道 | 订单量下降 | -18.2% |
| 华东重点产品 | 转化率下降 | -4.1pp |

判断：问题不是平均分布，而是集中在特定渠道与特定产品。
解释：问题点越集中，后续越应优先查结构因素，而不是把所有维度都平铺展开。
```

## 深入段落怎么写
- `问题识别`：确认异常存在
- `原因分析`：解释最主要的 1-2 个成因
- `影响评估`：说明影响范围和严重程度
- `下一步动作`：只给和问题直接相关的动作

## 正文与附件边界
- 正文：问题识别表、关键原因证据块、影响评估
- 附件：全量异常对象、细分渠道明细、长周期补充对比

## 不要这样写
- 一开始就给建议，没有先把问题证据摆出来
- 把“原因分析”写成主观猜测
- 只说“问题较大”，不量化偏离程度
- 每段都引用 CSV，却没有正文表格
